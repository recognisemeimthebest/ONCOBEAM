1데이터 파이프라인

원본 엑셀(QIN-HEADNECK Batch1+2, 시트별 중복헤더 제거 후 278명)을 아주대 데이터설명서의 38컬럼 정형 스키마로 변환했다.

원본 엑셀 (.xlsx)→ transform_clinical_to_schema.py→ clinical_data_schema.csv (278×38)→ load_csv_to_db.py→ PostgreSQL · patients

주요 변환 규칙

  • locationcancer = 전부 4(두경부암), birth_date = 연도+가상월(YYYYMM)
  • treatmethod / treatech / relapse / deathsign = 코드 매핑
  • 방사선 단위 Gy 통일: cGy 혼재 정규화(총선량>200 ÷100, 회당>100 ÷100) → 분할횟수 재계산
  • 회당선량 오기재 5건(32/35/36 = 분할횟수 오기재) 추정 보정 → 회당 2Gy로 정정

4CDSS 모듈 명세

모듈 1 — HTE (이질적 처치효과)

처치 = treatmethod 4군. 출력 = 쌍대 처치효과 4종 (2×2 요인: 항암추가 · 수술추가).

처치군인원비교 대비
1 방사선단독241vs2, 3vs4 (항암 추가효과)
1vs3, 2vs4 (수술 추가효과)
2 항암방사선165
3 수술+방사선35
4 수술+항암방사선40

대기 결과변수(outcome)는 파생결과로 추후 제공 → 현재 구조만 준비.

모듈 2 — 예후예측

입력 = 임상 16종 (+ 라디오믹스). 출력 = 5년 무재발생존 이진 (1=5년내 재발·사망 없음 / 0=있음).

라벨인원
1 (5년 무사건)87
0 (5년내 사건)115
모호(5년 전 중도절단, 제외)76
학습가능202
라디오믹스 확보 불가 → 모듈2는 임상변수 단독으로 구성. 결합 설계(RADIOMICS_FUSION.md: 2-stage Rad-score + ComBat)는 향후용 휴면 문서로 보존.

5서빙 · 모델

  • 단일 진실 공급원 cdss/features.py — 학습·서빙이 같은 feature 정의/인코딩 공유 (불일치=조용한 오예측 방지)
  • 입력 feature 17종 + 라벨 함수(rfs_label_5yr, survival_label) + 처치군(treatment_arm, HTE_CONTRASTS)
  • 모델 래퍼(model_wrappers.py)에 imputer 내장 → pkl 하나로 전처리+추론, 학습↔서빙 호환
  • 모델이 없으면 더미 응답 반환 → 모델 없이도 전체 흐름 동작
프로젝트 성격: 모델은 외부에서 완성(train/val). 이 레포의 ml_models/*.joblib은 외부 pkl 도착 전까지의 임시 스탠드인, ml_training/*은 참고용. 외부 pkl 수령 시 입력명세·전처리 포함여부에 맞춰 features.py 정렬 후 교체. 파생변수는 규칙을 받으면 서빙에서 자동 생성 가능.

참고 베이스라인(5-fold CV): 예후 Cox C-index ≈ 0.64, HTE 결과모델 AUC ≈ 0.69 (학습데이터 기준 0.70 → 정직한 일반화 추정).

6운영 테이블 + 예측 로깅

예측 API 호출 시 결과와 접근 이력이 자동 기록된다.

POST /api/cdss/hte · /prognosis→ features 생성→ 모델 예측→ Prediction + AuditLog 기록
  • predictions: 환자(정수 id)·모듈·입력스냅샷(JSON)·출력(JSON)·모델버전·사용자·시각
  • model_registry: pkl 버전·파일명·입력명세·외부지표·활성여부
  • audit_log: 사용자·action·resource·시각

검증: 환자기반 예측(patient_id=557)·features기반(patient_id=None) 모두 기록 확인.
